# Multi-Locale Setup (Optional)

The source site runs 27 locales with AI-translated blog posts. If the target site wants this, here's what needs to happen beyond the default single-language setup.

## Prerequisites

- You chose the i18n variant during Phase 2 (kept `middleware.i18n.ts` as `middleware.ts`, kept `env.d.i18n.ts` as `env.d.ts`)
- Migration `006_post_translations.sql` has been applied

## Required files: `i18n/config.ts` + `i18n/utils.ts`

These aren't bundled because locale lists are site-specific. Create them:

### `src/i18n/config.ts`

```typescript
export const locales = ['en', 'es', 'fr', 'de'] as const; // expand to your list
export const defaultLocale = 'en';
export type Locale = typeof locales[number];
```

### `src/i18n/utils.ts`

```typescript
import { locales, defaultLocale, type Locale } from './config';

export function getLocaleFromPath(path: string): Locale {
  const seg = path.split('/')[1];
  if (locales.includes(seg as Locale)) return seg as Locale;
  return defaultLocale;
}

// Minimal translation helper. Extend with a dictionary file per locale.
export function t(locale: Locale, key: string): string {
  return key; // TODO: replace with lookup table
}
```

For the source site, `i18n/utils.ts` also exports an `isLocalizedPath` helper and a `t()` that reads from `src/i18n/translations/{locale}.ts` dictionaries. Model yours after that if you want full localization.

## Locale-prefixed routes

Create `src/pages/[locale]/blog.astro` and `src/pages/[locale]/blog/[slug].astro` that render `<BlogPage />` and `<BlogPostPage />`. Both components already call `getPublishedPostsLocalized` and `getPostBySlugLocalized`, which overlay translations from `post_translations` when `locale !== 'en'`.

Add `getStaticPaths` returning each locale, or set `export const prerender = false` and let middleware route `/es/blog/foo` to `src/pages/[locale]/blog/[slug].astro`.

## Translating posts

The skill bundles `src/lib/translate.ts` and `src/pages/api/admin/cms/posts/[id]/translate.ts`. When an admin clicks "Translate" on a published post, the API:

1. Fetches the English post content
2. Sends it to OpenRouter (model: `google/gemini-3.1-flash-lite-preview`) with a schema-enforced prompt
3. Generates translated title, slug, excerpt, meta, content, FAQs for each target locale (4 in parallel)
4. Upserts into `post_translations`

### Config

- `OPENROUTER_API_KEY` must be set (env or secret)
- `src/lib/translate.ts` has a hardcoded list of locales at the top — update to match your `locales` array
- The prompt in `translate.ts` mentions the source site's topic ("image hosting") — edit for the target site

### Cost

Gemini 3.1 Flash Lite is cheap (~$0.075 / 1M input tokens). A typical 2000-word post × 26 target locales ≈ $0.10 per translated post. Bulk translation of a 100-post blog runs ~$10.

### Slug collisions

The translate function appends the post ID suffix if a translated slug already exists in that locale. Stored in `post_translations.slug`. `getPostBySlugLocalized` looks up by `(locale, slug)` so collisions don't cause 404s — they just produce longer slugs.

## Language switcher on blog post pages

`BlogPostPage.astro` needs to know every locale's slug for this post to render hreflang links. Query all translations:

```typescript
const translations = await db
  .prepare('SELECT locale, slug FROM post_translations WHERE post_id = ?')
  .bind(post.id)
  .all();

const hreflangAlternates = (translations.results || []).map(t => ({
  locale: t.locale,
  url: `https://{{DOMAIN}}/${t.locale}/blog/${t.slug}`,
}));
// + always add the English one
hreflangAlternates.push({ locale: 'en', url: `https://{{DOMAIN}}/en/blog/${post.slug}` });
```

Pass `hreflangAlternates` into the Layout — it's already a supported prop in the source site's Layout.astro.

## When NOT to do i18n

- Single-language sites — just skip everything here. The default `middleware.ts` + `getPublishedPosts` (non-localized) work fine.
- Sites with <3 languages where manual translation is cheaper than OpenRouter calls — keep the `post_translations` table but skip the `translate.ts` AI flow. Translate manually via the admin UI by editing each locale's row.

## Debugging

- "Translation missing, seeing English content": fallback is working correctly. Check `post_translations` for a row with your `(locale, post_id)`.
- "Slug not found 404 on /es/blog/mi-post": lookup is `(locale, slug)`. Confirm `post_translations.slug` has the exact Spanish slug.
- Translation API fails: check OpenRouter dashboard for rate limits or invalid JSON schema errors. `translate.ts` retries once on parse failure and falls back to English content.
